üìñ Introduction

What is Physical Design?

Physical design is the process of transforming a circuit description (netlist) into a physical layout that can be fabricated on silicon. It bridges the gap between logical design and manufacturing.

Physical design converts RTL (Register Transfer Level) code into actual chip geometry that can be manufactured.

What is Synopsys Fusion Compiler?

Fusion Compiler is Synopsys' next-generation RTL-to-GDSII implementation tool that combines synthesis, place and route, and signoff into a unified platform. It offers superior quality of results (QoR) with faster turnaround time.

2. Floorplanning

Definition

Floorplanning is the process of determining the size, shape, and location of modules and macros in a chip, along with the I/O pad placement.

Key Goals: Minimize chip area, reduce wire length, meet timing constraints, optimize power distribution, and ensure routability.

Important Floorplanning Parameters

Parameter Description Typical Value
Core Utilization % of core area occupied by cells 60-80%
Aspect Ratio Height/Width ratio 1.0 (square preferred)
Core-to-Die Spacing Space between core and die edge 10-50 ¬µm
Row Height Standard cell row height Multiple of library height
Warning: Too high core utilization (>85%) can lead to routing congestion and timing closure issues. Leave enough white space for optimization!

3. Placement

Definition

Placement is the process of assigning exact physical locations to all standard cells, macros, and other components on the chip while optimizing for timing, power, and routability.

Types of Placement

üîπ Global Placement

Coarse placement that determines approximate locations of cells. May have overlaps. Focuses on overall optimization.

üîπ Legalization

Removes overlaps from global placement and aligns cells to legal grid positions (rows).

üîπ Detailed Placement

Fine-tunes cell positions to optimize local connections, reduce wire length, and improve timing.

Placement Objectives: Minimize total wire length, meet timing constraints, reduce power consumption, avoid routing congestion, and maintain thermal uniformity.

4. Clock Tree Synthesis (CTS)

Definition

CTS is the process of distributing the clock signal from the clock source to all sequential elements (flip-flops) in the design with minimal skew and insertion delay.

Clock Skew: The difference in arrival time of the clock signal at different flip-flops. Minimize skew for reliable operation.

CTS Objectives

  • Minimize Clock Skew - Ensure all FFs receive clock simultaneously
  • Minimize Insertion Delay - Reduce clock latency
  • Balance Clock Tree - Equal path lengths to all sinks
  • Meet Slew/Capacitance Limits - Maintain signal integrity
  • Reduce Power Consumption - Clock network consumes 30-40% power
Warning: Don't perform CTS too early! Complete placement and pre-CTS optimization first. Clock tree is hard to modify once built.

5. Routing

Definition

Routing is the process of creating physical connections between cells using metal layers according to the netlist, while following design rules and optimizing for performance.

Routing Stages

Stage Description Focus
Global Routing Assigns nets to routing regions Overall connectivity planning
Track Assignment Assigns nets to specific tracks Detailed path planning
Detail Routing Creates actual metal geometries DRC-clean routing
Search & Repair Fixes DRC violations Clean up and optimization
Metal Layer Usage: Lower layers (M1-M2) for local connections, middle layers (M3-M5) for signal routing, upper layers (M6+) for power/ground and long-distance signals.

6. Timing Analysis

Static Timing Analysis (STA)

STA verifies the timing performance of a design by checking all possible paths without requiring simulation. It ensures setup and hold time requirements are met.

Critical Timing Concepts

Concept Definition Constraint
Setup Time Min time data must be stable before clock edge Data must arrive early enough
Hold Time Min time data must be stable after clock edge Data must stay stable long enough
Slack Difference between required and arrival time Must be positive (‚â•0)
Critical Path Path with worst (minimum) slack Focus optimization here
Common Timing Violations:
‚ö†Ô∏è Setup Violation: Data arrives too late ‚Üí Reduce logic depth, upsize cells, optimize placement
‚ö†Ô∏è Hold Violation: Data changes too early ‚Üí Add delay buffers, increase wire length

7. Design Rule Check (DRC) & Layout Versus Schematic (LVS)

Design Rule Check (DRC)

DRC verifies that the layout meets all manufacturing rules provided by the foundry, including minimum width, spacing, enclosure, and density rules.

Layout Versus Schematic (LVS)

LVS ensures that the physical layout exactly matches the circuit schematic/netlist. It verifies connectivity and device properties.

Check Type Purpose Tool
DRC Manufacturability verification Calibre, ICV, PVS
LVS Functionality verification Calibre nmLVS, Assura
Antenna Check Prevent gate oxide damage Built into DRC deck
ERC Electrical rule checking Integrated in LVS
Zero DRC/LVS violations required for tape-out! These are mandatory signoff checks.

üîÑ Complete Physical Design Flow

Step-by-Step Process

Phase 1: Design Setup

Inputs Required:
‚Ä¢ Gate-level netlist (.v file)
‚Ä¢ Technology library (.db files)
‚Ä¢ Physical library (LEF files)
‚Ä¢ Timing constraints (.sdc file)
‚Ä¢ Floorplan specifications
‚Ä¢ Power intent (UPF file - optional)

Phase 2: Floorplanning

  1. Initialize floorplan with dimensions
  2. Create power rings and straps
  3. Place macros and hard IPs
  4. Define I/O pad locations
  5. Create placement blockages if needed
  6. Verify floorplan quality

Phase 3: Power Planning

  1. Create VDD/VSS rings around core
  2. Add power straps across the core
  3. Connect straps to rings
  4. Place standard cell rails
  5. Verify power grid integrity

Phase 4: Placement

  1. Run coarse placement (global)
  2. Perform high-fanout net synthesis
  3. Optimize placement for timing
  4. Legalize placement
  5. Perform detailed placement
  6. Check congestion and timing

Phase 5: Clock Tree Synthesis

  1. Define clock tree specifications
  2. Identify clock sources and sinks
  3. Build clock tree structure
  4. Optimize for skew and insertion delay
  5. Route clock nets
  6. Verify clock quality

Phase 6: Routing

  1. Run global routing
  2. Track assignment
  3. Detailed routing
  4. Search and repair DRC violations
  5. Optimize routed design
  6. Perform ECO routing if needed

Phase 7: Signoff Verification

  1. Extract parasitic RC
  2. Run signoff STA
  3. Fix timing violations
  4. Run DRC and LVS
  5. Check antenna violations
  6. Generate final GDSII

‚ö° Fusion Compiler Unique Features

1. Unified RTL-to-GDSII Flow

Unlike traditional flows where synthesis (Design Compiler) and P&R (IC Compiler) are separate, Fusion Compiler integrates both in a single tool, eliminating handoff issues and providing better optimization.

2. Golden Signoff Integration

Built-in PrimeTime (timing), StarRC (extraction), and RedHawk (power) engines ensure signoff-quality results during implementation, reducing iterations.

3. Multi-Objective Concurrent Optimization

Simultaneously optimizes power, performance, and area (PPA) throughout the flow rather than in isolated stages.

4. Advanced Node Support

Native support for FinFET technologies (7nm, 5nm, 3nm) with multi-Vt optimization and advanced design rules.

Key Advantage: Fusion Compiler can achieve 10-20% better PPA compared to traditional DC + ICC2 flow with 2-3x faster runtime!

Setup Script (setup.tcl)

Main Fusion Compiler Script (fc_flow.tcl)

###################################### # Fusion Compiler Physical Design Flow ###################################### # 1. START FUSION COMPILER AND LOAD DESIGN source setup.tcl # Read netlist read_verilog $netlist link_design $design_name # Read constraints read_sdc $sdc_file # Load physical libraries read_lef "./lef/tech.lef" read_lef "./lef/cells.lef" # Set TLU+ models for RC extraction set_tlu_plus_files \ -max_tluplus $tlu_max \ -min_tluplus $tlu_min \ -tech2itf_map $tech_map # 2. FLOORPLANNING initialize_floorplan \ -core_utilization 0.7 \ -core_offset {10 10 10 10} \ -flip_first_row # Create power rings create_pg_ring_pattern ring_pattern \ -horizontal_layer M5 \ -horizontal_width 2 \ -horizontal_spacing 1 \ -vertical_layer M6 \ -vertical_width 2 \ -vertical_spacing 1 set_pg_strategy core_ring \ -core \ -pattern {{name: ring_pattern} {nets: {VDD VSS}}} # Create power mesh create_pg_mesh_pattern mesh_pattern \ -layers {{vertical_layer: M5} {width: 0.5} {spacing: 2} {pitch: 20}} \ -layers {{horizontal_layer: M6} {width: 0.5} {spacing: 2} {pitch: 20}} set_pg_strategy core_mesh \ -core \ -pattern {{name: mesh_pattern} {nets: {VDD VSS}}} # Apply power strategies compile_pg -strategies {core_ring core_mesh} # Connect standard cell rails create_pg_std_cell_conn_pattern rail_pattern \ -layers M1 set_pg_strategy rail_strat \ -core \ -pattern {{name: rail_pattern} {nets: VDD VSS}} compile_pg -strategies rail_strat # Save floorplan checkpoint save_block -as floorplan # 3. PLACEMENT # Set placement options set_app_options -name place.coarse.continue_on_missing_scandef -value true # Run placement place_opt # Check placement quality report_qor report_congestion report_timing -max_paths 10 # Save placement checkpoint save_block -as placed # 4. CLOCK TREE SYNTHESIS # Set CTS options set_app_options -name cts.common.max_fanout -value 32 set_app_options -name cts.compile.enable_global_route -value true # Define clock tree specifications set_clock_tree_options \ -target_skew 0.1 \ -max_capacitance 0.3 # Run CTS clock_opt # Report clock tree quality report_clock_qor report_clock_timing -type skew # Save CTS checkpoint save_block -as cts # 5. ROUTING # Set routing options set_app_options -name route.common.verbose_level -value 2 set_app_options -name route.detail.eco_route_use_soft_spacing -value true # Run routing route_auto # Report routing quality report_qor report_timing # Save routed checkpoint save_block -as routed # 6. OPTIMIZATION AND SIGNOFF # Run post-route optimization route_opt # Check design check_design -checks pre_clock_tree_stage check_legality # Generate reports report_qor > reports/final_qor.rpt report_timing -max_paths 100 > reports/final_timing.rpt report_power > reports/final_power.rpt report_area > reports/final_area.rpt # 7. STREAM OUT GDSII # Define layer map set_write_stream_options \ -map_layer "./tech/gds.map" \ -child_depth 0 # Write GDSII file write_stream \ -format gds \ -lib_name $design_name \ "./output/${design_name}.gds" # Write final netlist write_verilog "./output/${design_name}_final.v" # Write SDF for timing simulation write_sdf "./output/${design_name}.sdf" puts "Physical design flow completed successfully!"

‚ùì Expected Exam Questions

What is the purpose of physical design in VLSI?
Physical design transforms a gate-level netlist into a physical layout (GDSII) that can be manufactured. It involves floorplanning, placement, routing, and verification to create a design that meets timing, power, and area requirements while being manufacturable.
Explain the difference between global placement and detailed placement.
Global placement determines approximate locations of cells without considering legality (cells may overlap). It focuses on overall optimization goals. Detailed placement refines these positions to legal locations (no overlaps, aligned to rows) and optimizes local connectivity and timing.
What is clock skew and why is it important?
Clock skew is the difference in arrival time of the clock signal at different sequential elements (flip-flops). It's critical because excessive skew can cause timing violations. Positive skew can help setup time but hurt hold time, while negative skew does the opposite. Minimizing skew ensures reliable synchronous operation.
What are the typical core utilization values and why can't we use 100%?
Typical core utilization ranges from 60-80%. We can't use 100% because: (1) cells need space for routing channels, (2) optimization requires flexibility to move/resize cells, (3) high utilization causes routing congestion, (4) some whitespace is needed for buffers and optimization, (5) it impacts yield and manufacturability.
Explain the difference between DRC and LVS.
DRC (Design Rule Check) verifies the layout meets manufacturing rules like minimum width, spacing, and density - ensures manufacturability. LVS (Layout Versus Schematic) verifies the layout connectivity matches the schematic/netlist - ensures functional correctness. Both must pass for tape-out.
What is the difference between setup time and hold time violations?
Setup violation occurs when data arrives too late (after required time before clock edge) - path is too slow. Fix by reducing logic depth, upsizing cells, or optimizing placement. Hold violation occurs when data changes too early (before required time after clock edge) - path is too fast. Fix by adding delay buffers or increasing wire length.
What are the advantages of Fusion Compiler over traditional DC+ICC2 flow?
Fusion Compiler advantages: (1) Unified RTL-to-GDSII platform eliminates handoff issues, (2) Golden signoff integration reduces iterations, (3) Concurrent multi-objective optimization for better PPA, (4) 10-20% better QoR, (5) 2-3x faster runtime, (6) Native support for advanced nodes (FinFET).
What is the purpose of TLU+ files in physical design?
TLU+ (Table Look-Up Plus) files contain resistance and capacitance tables for accurate parasitic extraction based on physical layout geometry. They model RC parasitics for different wire widths, spacings, and layers, enabling accurate timing analysis during physical implementation.
Explain the routing stages in order.
Routing stages: (1) Global Routing - assigns nets to routing regions/tiles, creates routing guides, (2) Track Assignment - assigns nets to specific tracks within regions, (3) Detail Routing - creates actual metal geometries on specific layers and tracks, (4) Search & Repair - fixes DRC violations and optimizes routes.
What is the purpose of power rings and power mesh?
Power rings surround the core to provide robust VDD/VSS distribution from pads. Power mesh (straps) distributes power across the core area, reducing IR drop and ensuring uniform voltage supply. Together they form a low-resistance power distribution network that prevents voltage droop during switching.
What information is contained in LEF files?
LEF (Library Exchange Format) files contain physical information about standard cells and technology: (1) Cell dimensions and pin locations, (2) Metal layer definitions and routing rules, (3) Blockage information, (4) Site definitions for cell placement, (5) Via definitions. LEF is used for P&R while detailed transistor info is hidden.
Why is congestion analysis important during placement?
Congestion analysis identifies areas where routing demand exceeds available tracks. High congestion leads to: (1) Routing failures/DRCs, (2) Longer wire detours increasing delay, (3) Difficulty meeting timing, (4) More vias and complexity. Detecting congestion during placement allows cell spreading before routing.
What is the purpose of the SDC file in physical design?
SDC (Synopsys Design Constraints) file specifies timing constraints for the design: clock definitions, input/output delays, multicycle paths, false paths, load capacitances, drive strengths, max transition/capacitance limits. It guides optimization and enables timing verification (STA).
Explain the concept of timing slack.
Slack = Required Time - Arrival Time. Positive slack (>0) means timing is met with margin. Negative slack (<0) indicates timing violation. Setup slack uses latest arrival and earliest required time. Hold slack uses earliest arrival and latest required time. Critical path has the worst (minimum) slack.
What is the final output of physical design flow?
Final outputs include: (1) GDSII file - complete layout geometry for fabrication, (2) Final netlist - post-layout netlist with ECOs, (3) SDF file - timing delays for gate-level simulation, (4) Reports - timing, power, area QoR, (5) DRC/LVS clean status - verification sign-off.

‚ö†Ô∏è Common Mistakes to Avoid

Mistake 1: Incorrect Library Loading

Wrong

# Missing link library set target_library "slow.db" # Will cause link errors!

Correct

# Include * for memory set target_library "slow.db" set link_library "* slow.db fast.db"

Mistake 2: Too High Core Utilization

Wrong

# Will cause congestion! initialize_floorplan \ -core_utilization 0.95

Leads to routing congestion, no space for optimization, timing closure issues.

Correct

# Leave room for routing! initialize_floorplan \ -core_utilization 0.70

Provides adequate whitespace for routing and optimization.

Mistake 3: Running CTS Too Early

Wrong

Running CTS immediately after floorplan without placement optimization.

initialize_floorplan clock_opt # Too early!

Correct

Complete placement first, then run CTS.

initialize_floorplan place_opt clock_opt # After placement!

Mistake 4: Forgetting to Save Checkpoints

Wrong

Running entire flow without saving intermediate states.

place_opt clock_opt route_auto # No saves - if error, restart all!

Correct

Save after each major step for recovery.

place_opt save_block -as placed clock_opt save_block -as cts route_auto save_block -as routed

Mistake 5: Missing TLU+ Files Setup

Wrong

Not loading parasitic models leads to inaccurate timing.

# No TLU+ files loaded place_opt # Timing will be wildly optimistic!

Correct

Load TLU+ files before any optimization.

set_tlu_plus_files \ -max_tluplus max.tluplus \ -min_tluplus min.tluplus \ -tech2itf_map tech.map place_opt

Mistake 6: Ignoring Congestion Warnings

Wrong

Proceeding to routing despite high congestion warnings from placement.

Result: Routing DRCs, timing violations, failed design.

Correct

Check congestion after placement and fix before proceeding.

place_opt report_congestion # If high congestion: # - Reduce utilization # - Spread cells manually # - Add routing blockages

Mistake 7: Incorrect Clock Definitions

Wrong

# Period in ns but constraints in ps create_clock -period 10000 [get_ports clk] # Unit mismatch!

Correct

# Consistent units (ns) create_clock -period 10 [get_ports clk] # 10ns = 100MHz
