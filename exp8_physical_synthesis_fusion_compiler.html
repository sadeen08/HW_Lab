üí° Core Concepts

What is Physical Synthesis?

Physical synthesis is the process of transforming a gate-level netlist into a physical layout with actual geometries. It includes:

  • Floorplanning: Defining chip area and macro placement
  • Placement: Positioning standard cells in rows
  • Clock Tree Synthesis: Building balanced clock distribution network
  • Routing: Creating metal interconnections between cells
  • Optimization: Meeting timing, power, and area goals

Design Flow Stages

Stage Purpose Key Outputs
Floorplanning Define chip dimensions and plan macro locations Core area, aspect ratio, utilization
Placement Position standard cells optimally Cell locations, congestion map
CTS Build clock distribution network Clock tree, skew, latency
Routing Create physical metal connections Routed design, DRC report
Sign-off Final verification and tape-out prep GDSII, timing reports

üìê Floorplanning

What is Floorplanning?

Floorplanning is the first physical design step where you define the chip's dimensions, allocate space for cells, and plan macro block locations. It sets the foundation for successful placement and routing.

Key Floorplanning Parameters

Parameter Definition Typical Range Impact
Core Utilization % of core area occupied by cells 60-80% Higher = denser, harder to route
Aspect Ratio Height / Width ratio 0.5 - 2.0 1.0 = square, affects wire length
Core-to-IO Space between core and pads 10-50 Œºm Room for routing and power
Row Spacing Gap between cell rows Site height Affects routing channels
Utilization Trade-off: 70% utilization is often optimal - leaves room for routing while maintaining reasonable area. Too high (>85%) causes congestion; too low (<50%) wastes area.

Floorplanning Commands in Fusion Compiler

# Initialize floorplan with dimensions initialize_floorplan \ -core_utilization 0.7 \ -core_aspect_ratio 1.0 \ -core_offset {10 10 10 10} \ -flip_first_row # Or specify exact dimensions initialize_floorplan \ -core_width 500 \ -core_height 500 \ -left_io2core 20 \ -right_io2core 20 \ -top_io2core 20 \ -bottom_io2core 20 # View the floorplan gui_start
Common Issue: If utilization is too high, you'll see "Cannot place all cells" or routing congestion errors. Solution: Reduce utilization or increase core area.

Analyzing Floorplan Quality

  • ‚úì Check core area vs. cell area match expected utilization
  • ‚úì Verify aspect ratio is reasonable for your design
  • ‚úì Ensure adequate space for routing (30-40% free)
  • ‚úì Check pin placement allows good connectivity
  • ‚úì Verify power ring can be placed around core

üìç Placement

What is Placement?

Placement is the process of determining physical locations for all standard cells within the floorplan. Good placement minimizes wire length, reduces congestion, and helps meet timing constraints.

Placement Stages

Stage Purpose Characteristics
Coarse Placement Initial rough positioning Fast, allows overlap, establishes general layout
Legalization Make placement legal Remove overlaps, align to rows, respect blockages
Detail Placement Fine-tune cell positions Optimize timing, minimize wire length
Post-Placement Opt Fix timing violations Buffer insertion, gate sizing, cell swapping

Placement Commands

# Basic placement flow create_placement -effort high # Legalize placement (remove overlaps) legalize_placement # Optimize timing after placement place_opt # Generate placement report report_placement -physical report_congestion report_timing -max_paths 10 # Check for design rule violations check_legality
Placement Quality Metrics: Look for low congestion (<20%), good timing (positive slack), and even cell distribution. High congestion areas may need floorplan adjustments.

Placement Optimization Techniques

üîÑ Buffer Insertion

Add buffers on long nets to fix timing and improve signal integrity

üìè Gate Sizing

Upsize/downsize cells to balance timing vs. power/area

‚ö° Vt Swapping

Change cell threshold voltage (LVT/HVT) for speed/leakage tradeoff

üéØ Pin Assignment

Optimize pin locations to minimize routing distance

Watch Out: If placement congestion exceeds 80% in any region, routing will likely fail. Consider reducing utilization or using placement blockages to guide cell distribution.

Analyzing Placement Results

Check Command Good Value
Cell Density report_placement Even distribution, no hotspots
Congestion report_congestion < 20% overflow
Timing Slack report_timing WNS > 0, TNS = 0
Legality check_legality No overlaps or violations

üïê Clock Tree Synthesis (CTS)

What is Clock Tree Synthesis?

CTS builds a balanced clock distribution network that delivers the clock signal from the clock source to all sequential elements (flip-flops, registers) with minimal skew and acceptable latency.

Key CTS Concepts

Term Definition Goal
Clock Skew Difference in arrival time between clock pins Minimize (< 50-100 ps)
Clock Latency Delay from clock source to registers Balance across tree
Insertion Delay Total delay through clock tree Minimize while meeting skew
Clock Transition Rise/fall time of clock signal < 10% of clock period
Fanout Number of loads per buffer Balance (typically 8-16)
Skew vs. Latency: Low skew is critical for timing - it ensures all flip-flops receive clock edges simultaneously. Latency can be higher as long as it's balanced and doesn't violate setup/hold times.

Clock Tree Topologies

üå≥ H-Tree

Symmetric tree structure, excellent skew, higher wire length

üéã X-Tree

Diagonal routing, good for square floorplans

üå≤ Fishbone

Spine with branches, good for rectangular designs

üîÄ Hybrid

Combines topologies, optimized for specific designs

CTS Commands in Fusion Compiler

# Set CTS constraints set_clock_tree_options \ -target_skew 0.1 \ -max_transition 0.2 \ -max_capacitance 0.5 # Specify clock tree reference cells (buffers/inverters) set_lib_cell_purpose -include cts [get_lib_cells */*BUF*] set_lib_cell_purpose -include cts [get_lib_cells */*INV*] # Build clock tree clock_opt -from build_clock -to route_clock # Or step by step: synthesize_clock_trees route_clock_trees optimize_clock_trees # Generate CTS reports report_clock_tree -summary report_clock_timing -type skew report_clock_timing -type latency
CTS Failure: If CTS fails with "Unable to meet skew target", try: 1) Increase target skew, 2) Allow more CTS buffer types, 3) Reduce clock frequency, or 4) Improve placement quality.

CTS Quality Checks

Metric Check Command Acceptable Range
Global Skew report_clock_timing -type skew < 5% of clock period
Max Insertion Delay report_clock_tree -summary < 30% of clock period
Clock Transition report_constraint -all_violators < 10% of clock period
Buffer Levels report_clock_tree Balanced (within 1 level)

üõ£Ô∏è Routing

What is Routing?

Routing creates the physical metal interconnections between all pins using available metal layers. It must satisfy design rules, minimize wire length, avoid congestion, and meet electrical constraints.

Routing Stages

Stage Description Purpose
Global Routing Assigns nets to routing regions Plan overall routing strategy, identify congestion
Track Assignment Assigns wires to specific tracks Allocate routing resources within regions
Detail Routing Creates actual metal geometries Complete all connections with real wires
Search & Repair Fixes DRC violations Resolve spacing, width, via violations

Metal Layer Stack

Layer Direction Typical Use Width/Pitch
Metal 1 (M1) Horizontal Cell-level routing Narrow (~0.1 Œºm)
Metal 2 (M2) Vertical Local connections Narrow (~0.1 Œºm)
Metal 3-4 Horizontal/Vertical Intermediate routing Medium (~0.2 Œºm)
Metal 5-6+ Horizontal/Vertical Global routing, power Wide (~0.5+ Œºm)

üí° Lower layers (M1-M2) have higher resistance and are used for short connections. Upper layers are for long-distance routing and power distribution.

Routing Direction: Alternating horizontal/vertical layers reduces shorts and simplifies via placement. M1 horizontal, M2 vertical, M3 horizontal, etc.

Routing Commands

# Set routing options set_route_opt_options \ -optimize_wire_via_effort_level high \ -crosstalk_driven true # Route the design route_opt # Or step by step: route_global route_track route_detail # Fix routing violations route_search_repair # Optimize after routing route_opt -incremental # Check for violations check_routes verify_drc report_route -verbose
DRC Violations: Common types include minimum spacing, minimum width, via enclosure, and short circuits. Use verify_drc and route_search_repair iteratively to fix them.

Design Rule Check (DRC) Categories

üìè Spacing Violations

Wires too close together on same layer

üìê Width Violations

Wires narrower than minimum allowed

‚ö° Short Circuits

Different nets touching illegally

üîå Via Violations

Improper via enclosure or spacing

Post-Route Analysis

Analysis Command What to Check
Route Completion report_route 100% nets routed, no opens
DRC Violations verify_drc Zero violations
Timing report_timing WNS > 0, all paths meet timing
Congestion report_congestion No overflow regions
Wire Length report_design -physical Reasonable total wire length

Full Fusion Compiler Script - Setup Phase

# ============================================ # Fusion Compiler - Physical Synthesis Script # ============================================ # Set up library search path set_app_var search_path "./libs ./designs $search_path" # Define target library (timing/power data) set_app_var target_library "slow.db" # Define link library (all libraries used) set_app_var link_library "* slow.db fast.db" # Technology file (LEF - physical layout) read_lef "tech.lef" read_lef "stdcells.lef" # Read synthesized netlist read_verilog "design.v" # Set current design current_design top_module # Link the design (resolve references) link_design # Read timing constraints read_sdc "design.sdc" # Save design database save_block -as design_setup

Floorplanning Script

# ============================================ # STEP 1: Floorplanning # ============================================ # Initialize floorplan with utilization initialize_floorplan \ -core_utilization 0.7 \ -core_aspect_ratio 1.0 \ -core_offset {10 10 10 10} \ -flip_first_row # Alternative: specify exact dimensions # initialize_floorplan \ # -core_width 500 \ # -core_height 500 \ # -left_io2core 20 \ # -right_io2core 20 \ # -top_io2core 20 \ # -bottom_io2core 20 # Create power/ground rings create_pg_ring_pattern ring_pattern \ -layers {M6 M7} \ -widths {2.0 2.0} \ -spacings {1.0 1.0} \ -core_offset {2.0 2.0} create_pg_ring \ -pattern ring_pattern \ -nets {VDD VSS} # Create power stripes create_pg_mesh_pattern mesh_pattern \ -layers {M6 M7} \ -widths {1.0 1.0} \ -pitches {50.0 50.0} create_pg_mesh \ -pattern mesh_pattern \ -nets {VDD VSS} # Connect power to standard cells create_pg_std_cell_conn_pattern cell_pattern \ -layers {M1} create_pg_std_cell_conn \ -pattern cell_pattern \ -nets {VDD VSS} # Report floorplan statistics report_floorplan report_design -physical # Save floorplan save_block -as design_floorplan

Placement Script

# ============================================ # STEP 2: Placement # ============================================ # Set placement options set_app_var place.coarse.continue_on_missing_scandef true # Perform coarse placement create_placement -effort high -timing_driven # Legalize placement (remove overlaps) legalize_placement # Check placement legality check_legality -verbose # Optimize placement for timing place_opt # Generate placement reports report_placement -physical report_congestion -grc_based report_timing -max_paths 10 -path_type summary report_qor -summary # Check for timing violations report_constraint -all_violators -nosplit # Save placed design save_block -as design_placed

Clock Tree Synthesis Script

# ============================================ # STEP 3: Clock Tree Synthesis # ============================================ # Define CTS constraints set_clock_tree_options \ -target_skew 0.1 \ -target_latency 1.0 \ -max_transition 0.2 \ -max_capacitance 0.5 \ -buffer_relocation true \ -gate_relocation true # Specify clock tree reference cells set_lib_cell_purpose -include cts \ [get_lib_cells */*CLKBUF*] set_lib_cell_purpose -include cts \ [get_lib_cells */*CLKINV*] # Exclude certain cells if needed set_lib_cell_purpose -exclude cts \ [get_lib_cells */*BUF_X1] # Build and optimize clock tree clock_opt -from build_clock -to route_clock # Alternative: step-by-step approach # synthesize_clock_trees # route_clock_trees # optimize_clock_trees # Generate CTS reports report_clock_tree -summary report_clock_tree -structure report_clock_timing -type skew -verbose report_clock_timing -type latency -verbose report_timing -delay_type max -max_paths 5 # Check for clock violations report_constraint -all_violators -clock # Save CTS design save_block -as design_cts

Routing Script

# ============================================ # STEP 4: Routing # ============================================ # Set routing options set_route_opt_options \ -optimize_wire_via_effort_level high \ -setup_margin 0.05 \ -hold_margin 0.05 # Enable crosstalk prevention set_route_opt_options -crosstalk_driven true # Perform global and detail routing with optimization route_opt # Alternative: step-by-step routing # route_global # Assign nets to routing regions # route_track # Assign to specific tracks # route_detail # Create actual wires # Fix routing violations route_search_repair -max_iterations 10 # Incremental optimization route_opt -incremental -size_only # Check routing quality check_routes # Verify design rules verify_drc # Generate routing reports report_route -verbose report_design -physical report_timing -delay_type max -max_paths 20 report_timing -delay_type min -max_paths 20 report_constraint -all_violators # Power analysis report_power -hierarchy # Area report report_area -hierarchy # Save routed design save_block -as design_routed

Final Sign-off and Output Generation

# ============================================ # STEP 5: Sign-off and Output Generation # ============================================ # Add filler cells (fill empty spaces) create_stdcell_filler \ -lib_cells {FILL1 FILL2 FILL4 FILL8} # Connect filler cells to power/ground connect_pg_net -net VDD [get_pins -hier */VDD] connect_pg_net -net VSS [get_pins -hier */VSS] # Final optimization route_opt -incremental -only_hold_time # Generate comprehensive reports report_qor > reports/final_qor.rpt report_timing -path_type summary > reports/final_timing.rpt report_power > reports/final_power.rpt report_area > reports/final_area.rpt report_clock_tree -summary > reports/final_clock.rpt # Write output files # 1. Verilog netlist (with physical info) write_verilog outputs/design_final.v # 2. DEF file (physical layout) write_def outputs/design_final.def # 3. GDSII file (for fabrication) write_gds outputs/design_final.gds # 4. SDC file (updated constraints) write_sdc outputs/design_final.sdc # 5. SPEF file (parasitics) write_parasitics -format SPEF \ -output outputs/design_final.spef # 6. SDF file (delay annotation) write_sdf outputs/design_final.sdf # Save final design save_block -as design_final puts "Physical synthesis complete!" exit

‚ùì Expected Exam Questions

What is the difference between logical synthesis and physical synthesis?
Logical synthesis converts RTL to gate-level netlist without considering physical layout (no placement/routing). Physical synthesis includes placement, routing, and physical optimization while meeting timing, considering actual wire delays and physical constraints.
Why is clock skew important and how does CTS minimize it?
Clock skew is the difference in clock arrival times at different flip-flops. Excessive skew can cause setup/hold violations. CTS minimizes skew by: 1) Building balanced tree structures (H-tree, X-tree), 2) Using matched buffer chains, 3) Equalizing wire lengths, 4) Buffer sizing to balance delays.
What does core utilization of 70% mean? What happens if it's too high or too low?
70% utilization means 70% of core area is occupied by standard cells, leaving 30% for routing. Too high (>85%) causes routing congestion and placement failures. Too low (<50%) wastes chip area and increases wire lengths, hurting timing and power.
Explain the metal layer stack and why layers have alternating directions.
Metal layers are stacked vertically with alternating routing directions (M1 horizontal, M2 vertical, M3 horizontal, etc.). This: 1) Reduces shorts between parallel wires, 2) Simplifies via connections (only need to change one coordinate), 3) Enables efficient maze routing algorithms, 4) Lower layers (M1-M2) for local routing, upper layers for global/power.
What are DRC violations and name at least 3 types?
DRC (Design Rule Check) violations are physical layout errors that violate manufacturing rules. Types: 1) Minimum spacing - wires too close on same layer, 2) Minimum width - wires too narrow, 3) Short circuits - different nets touching, 4) Via enclosure - insufficient metal around via, 5) Antenna violations - long metal causing charge buildup.
What is the purpose of floorplanning? What decisions are made?
Floorplanning defines the physical foundation of the chip. Decisions include: 1) Core dimensions and aspect ratio, 2) Utilization percentage, 3) Macro block placement, 4) I/O pad locations, 5) Power ring/mesh strategy, 6) Routing channel allocation. Good floorplanning prevents congestion and enables timing closure.
How does placement optimization improve timing?
Placement optimization improves timing through: 1) Buffer insertion - breaks long nets to reduce RC delay, 2) Gate sizing - upsize critical path cells for speed, downsize non-critical for power, 3) Cell spreading - reduce congestion to improve routability, 4) Vt swapping - use Low-Vt cells on critical paths for speed.
What input files are required for physical synthesis in Fusion Compiler?
Required files: 1) .lib/.db - Liberty timing libraries with cell delays, 2) .lef - Library Exchange Format with physical cell dimensions, 3) .tf - Technology file with process rules and metal layers, 4) .v - Synthesized gate-level netlist, 5) .sdc - Synopsys Design Constraints with timing requirements.
What is the difference between global routing and detail routing?
Global routing divides the chip into routing regions and assigns nets to regions, creating a coarse routing plan. It identifies congestion hotspots. Detail routing creates actual metal geometries on specific tracks within each region, following design rules and completing all connections with real wires and vias.
Explain the trade-off between clock skew and clock latency.
Skew must be minimized (typically <100ps) as it directly affects setup/hold margins and can cause violations. Latency can be higher as long as it's balanced across the tree - all flip-flops see the same delay. Adding buffers reduces skew but increases latency. The goal is zero skew with acceptable latency (<30% of clock period).
What happens during the legalization step in placement?
Legalization makes coarse placement legal by: 1) Removing cell overlaps by spreading cells apart, 2) Aligning cells to standard cell rows, 3) Respecting placement blockages and keep-out areas, 4) Orienting cells correctly (flipping for power rail alignment), 5) Ensuring minimum spacing between cells is maintained.
Why are filler cells added and what do they do?
Filler cells fill gaps between standard cells to: 1) Ensure continuous N-well and power/ground rails, 2) Prevent DRC violations from gaps, 3) Maintain uniform substrate biasing, 4) Improve manufacturing yield. They contain no logic, just power/ground connections and well layers.
What is the purpose of the SPEF file generated after routing?
SPEF (Standard Parasitic Exchange Format) contains extracted parasitic resistances and capacitances of all routed wires. It's used for: 1) Accurate post-route timing analysis, 2) Power analysis with real RC values, 3) Signal integrity analysis, 4) Back-annotation to simulation tools for verification.
Compare H-tree and Fishbone clock tree topologies.
H-tree: Symmetric, recursive tree with branches at 90¬∞. Pros: Excellent skew, balanced structure. Cons: Higher wire length, works best for square layouts. Fishbone: Spine with perpendicular branches. Pros: Lower wire length, good for rectangular designs. Cons: Slightly higher skew, less symmetric. Choice depends on floorplan shape.
What does route_opt do and when should it be used?
route_opt performs routing and simultaneous optimization. It: 1) Routes all nets while optimizing for timing, 2) Inserts buffers/gates during routing, 3) Performs layer assignment, 4) Fixes setup/hold violations, 5) Minimizes DRC violations. Use it as the main routing command for best PPA (Power, Performance, Area) results. Use incremental mode after initial routing for fine-tuning.

‚ö†Ô∏è Common Mistakes to Avoid

‚ùå Wrong

Setting utilization > 90%

Causes severe routing congestion, placement failures, and timing violations.

‚úÖ Correct

Use 60-80% utilization

Leaves adequate space for routing channels, optimization, and ECO changes.

‚ùå Wrong

Ignoring clock tree constraints

Running CTS without specifying target_skew or reference cells leads to poor clock quality.

‚úÖ Correct

Set CTS constraints properly

Define target_skew, max_transition, and explicitly list allowed CTS buffers/inverters.

‚ùå Wrong

Skipping legalization check

Proceeding to routing with illegal placement causes failures and wastes runtime.

‚úÖ Correct

Always run check_legality

Verify placement is legal before CTS/routing: check_legality -verbose

‚ùå Wrong

Not fixing DRC violations

Leaving design rule violations will fail fabrication and tapeout.

‚úÖ Correct

Iterate until zero DRCs

Use route_search_repair repeatedly and verify with verify_drc until clean.

‚ùå Wrong

Forgetting to save block

Long runtime stages lost if tool crashes without checkpoints.

‚úÖ Correct

Save after each major stage

Use save_block -as design_stagename after floorplan, placement, CTS, routing.

‚ùå Wrong

Using incorrect library files

Mixing libraries from different process nodes or corners causes errors.

‚úÖ Correct

Match all library files

Ensure .lib, .lef, and .tf are all from the same technology node and vendor.

‚ùå Wrong

Not checking timing after routing

Routing adds wire delays that can violate timing constraints.

‚úÖ Correct

Always verify post-route timing

Run report_timing for setup and hold after routing to ensure closure.
